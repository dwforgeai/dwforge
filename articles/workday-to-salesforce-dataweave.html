1. Why Workday to Salesforce sync is painful

Most HR integrations look straightforward on paper. Workday has employee records. Salesforce needs them. Connect the two. Done.

In reality it is one of the most frustrating integration patterns in enterprise MuleSoft work. Here is why:

  • Workday returns SOAP XML with deeply nested structures. Salesforce wants flat JSON.
  • Workday field names like wd:Worker_Data map to Salesforce fields like EmployeeID__c. Nothing is obvious.
  • HR data is full of optional fields. A terminated employee looks completely different from an active one.
  • Workday wraps everything in namespace prefixes. DataWeave handles this but you need to know the pattern.
  • Date formats do not match. Workday uses ISO 8601. Salesforce Date fields have their own expectations.

4. What Salesforce expects

Salesforce wants flat JSON. Clean field names. No namespaces. No nesting beyond what the object model requires.

{
  "EmployeeID__c": "EMP-10045",
  "FirstName": "Sarah",
  "LastName": "Mitchell",
  "Email": "sarah.mitchell@company.com",
  "Title": "Senior Integration Architect",
  "Department__c": "Integration Platform Team",
  "Location__c": "Melbourne AU",
  "IsActive__c": true,
  "HireDate__c": "2022-03-14",
  "TerminationDate__c": null,
  "StartDate__c": "2022-03-14"
}

6. XML to JSON in DataWeave

The first thing you need is the namespace declaration. Without this every wd: field returns null and you spend an hour debugging something that should take two minutes.

%dw 2.0
output application/json
ns wd http://workday.com/ns/bsvc/v40
---
// Now you can access Workday fields using the wd namespace
{
  EmployeeID__c: payload.wd#Worker.wd#Worker_Reference
    .wd#ID."#content" default ""
}
The namespace URL matters
The Workday namespace URL changes with API version. v40 above is an example. Check your actual Workday SOAP response for the correct namespace URL and match it exactly in your DataWeave script. A single character difference and every field returns null.

7. Null handling for HR fields

HR data is the messiest data in any enterprise. Fields that should always exist sometimes do not. Optional fields return empty strings instead of null. Workday Active field returns "1" or "0" as a string, not a boolean.

Converting Active status to boolean

IsActive__c: payload.wd#Worker.wd#Worker_Data
  .wd#Employment_Data.wd#Worker_Status_Data.wd#Active == "1"

Handling empty date fields

Workday sometimes returns an empty element for Termination_Date on active employees. You need to check for both null and empty string.

var terminationDateRaw = payload.wd#Worker.wd#Worker_Data
  .wd#Employment_Data.wd#Worker_Status_Data.wd#Termination_Date
---
{
  TerminationDate__c: if (terminationDateRaw != null and
    terminationDateRaw != "")
    terminationDateRaw as String
  else null
}

Defaulting missing strings

Title: payload.wd#Worker.wd#Worker_Data.wd#Employment_Data
  .wd#Worker_Job_Data.wd#Position_Data.wd#Position_Title default ""

8. Handling terminated employees

Terminated employee records need different handling. When Workday sends a termination the DataWeave needs to set IsActive__c to false and capture the termination date. Some teams also need to deactivate the Salesforce user record.

%dw 2.0
output application/json
ns wd http://workday.com/ns/bsvc/v40
var worker = payload.wd#Worker.wd#Worker_Data
var statusData = worker.wd#Employment_Data.wd#Worker_Status_Data
var isActive = statusData.wd#Active == "1"
var terminationDate = statusData.wd#Termination_Date
---
{
  IsActive__c: isActive,
  TerminationDate__c: if (!isActive and terminationDate != null
    and terminationDate != "")
    terminationDate as String
  else null,
  // Deactivate Salesforce user if terminated
  IsPortalEnabled: isActive
}

9. Batch sync transformation

In a scheduled sync the Workday response contains multiple Worker records inside a Get_Workers_Response wrapper. You need to map over the array.

%dw 2.0
output application/json
ns wd http://workday.com/ns/bsvc/v40
var workers = payload.wd#Get_Workers_Response
  .wd#Response_Data.wd#Worker
---
workers map (worker) -> do {
  var data = worker.wd#Worker_Data
  var personal = data.wd#Personal_Data
  var nameData = personal.wd#Name_Data
    .wd#Legal_Name_Data.wd#Name_Detail_Data
  var employment = data.wd#Employment_Data
  var jobData = employment.wd#Worker_Job_Data
  var statusData = employment.wd#Worker_Status_Data
  var orgData = data.wd#Organization_Data
    .wd#Worker_Organization_Data filter (org) ->
    (org.wd#Organization_Data.wd#Organization_Type_Reference
    .wd#ID == "DEPARTMENT")
  ---
  {
    EmployeeID__c: worker.wd#Worker_Reference.wd#ID."#content" default "",
    FirstName: nameData.wd#First_Name default "",
    LastName: nameData.wd#Last_Name default "",
    Email: personal.wd#Contact_Data.wd#Email_Address_Data
      .wd#Email_Address default "",
    Title: jobData.wd#Position_Data.wd#Position_Title default "",
    Department__c: orgData[0].wd#Organization_Data
      .wd#Organization_Name default "",
    Location__c: jobData.wd#Business_Site_Summary_Data.wd#Name default "",
    IsActive__c: statusData.wd#Active == "1",
    HireDate__c: statusData.wd#Hire_Date default null,
    TerminationDate__c: if (statusData.wd#Termination_Date != null
      and statusData.wd#Termination_Date != "")
      statusData.wd#Termination_Date as String
    else null
  }
}
